# AI Coding 协作问题与解决方向讨论

## 1. 核心问题

我在 AI Coding 使用中遇到的核心问题，并不是 AI 完全不会写代码，而是当 AI Coding 能力出现之后，用户对“Coding”这件事本身的工程认知没有同步跟上。

对于产品、运营或非研发背景的用户来说，AI Coding 可以很快把一个想法变成原型。但这个原型和真实研发团队想要交付的工程结果之间，往往存在差距。它可能能跑、能展示、能验证一个局部功能，但不一定能作为长期稳定、可维护、可上线的系统。

因此，AI Coding 当前容易停留在 Vibe Coding 阶段：一个人把自己的想法快速变成局部功能或原型。但企业真正需要的不是一次性的局部功能，而是能够被团队长期维护、复用、扩展和上线的工程系统。

除了工程协作问题之外，还有一个更具体的多模态使用痛点：Coding Agent 当前主要服务代码和文字模态，但真实 AI 产品工作里，经常需要处理图片、页面、报表、视频等视觉产物。

以 AI 生图业务为例，评测流程里需要批量查看图片结果、对比模板效果、标注 badcase、判断文字是否错误、画面是否可用、风格是否一致、是否符合商户上下文。IDE 或 Coding Agent 插件即使能预览单张图片，也很难承接这种批量视觉评测工作。因此很多时候需要临时写一个 HTML 工具，把图片、元数据、模板 id、质检结果组织成一个可浏览、可筛选、可标注的页面。

这个痛点可以概括为：

> Coding Agent 服务 Coding，但不服务人类的视觉判断；它能帮助生成代码和脚本，却没有很好承接“批量视觉产物验收”这类真实工作流。

所以这里存在两类问题：

- 多模态产物验收问题：AI 生成或处理的结果不只是代码，还包括图片、页面和批量视觉产物，但现有 Coding Agent 缺少高效的视觉预览、对比、标注和回流能力。
- 团队工程协作问题：AI Coding 可以快速做出局部原型，但要进入团队工程交付，还需要共识、规范、任务拆解、多人协同和验收机制。

## 2. Vibe Coding 和 Team Coding 的差异

Vibe Coding 更像是把一个人的想法变成原型。它关注的是局部功能是否能快速实现，是否能把一个想法用代码表达出来。

Team Coding 要解决的是另一件事：把团队共识变成可维护、可复用、可上线的工程结果。

工程可用背后包含大量隐性条件，例如：

- 公司或项目已有的代码习惯。
- 目录结构和技术栈约束。
- 字段结构、接口协议和数据约定。
- 设计规范和交互规范。
- 可靠性、可维护性和可扩展性。
- 与其他模块、系统或团队之间的依赖关系。

这些条件在传统研发流程中，往往通过需求评审、技术评审、代码评审、多人协作和反复对齐来补齐。AI Coding 如果绕过这些协作环节，就会很快，但也容易带来重复开发、局部最优、难合并、难维护的问题。

## 3. 复杂任务中的协作痛点

当任务比较简单时，一个人驱动一个 Agent 就可以完成。但当任务变复杂，需要多个 Agent 分别完成不同模块时，问题会更加明显。

复杂任务中的损耗主要体现在：

- 用户需要不断推动每个 Agent 执行任务。
- 用户需要在多个会话、终端、IDE、分支和文件之间切换。
- 每个 Agent 的目标、上下文、产物和当前状态难以统一管理。
- 不同 Agent 可能修改同一批文件，引发冲突。
- 各个局部功能看起来完成了，但组合起来不一定符合全局目标。
- 产物是否达到工程可用标准，需要额外评审和验收。

所以，真正的问题不是“如何让一个 Agent 更会写代码”，而是当一群 Agent 参与复杂工程任务时，如何让它们在共同目标、共享约束和清晰边界下协作。

## 4. 核心判断

AI Coding 的下一阶段，不只是一个人管理一个 AI，或者一个人管理一群 AI，而是要进入“一群人管理一群 AI”的协作模式。

这个模式下，人不再只是逐条 prompt 驱动 Agent 执行，也不只是管理文件系统里的最终产物，而是要管理一套 Agent 协作系统。

这套系统需要解决的问题包括：

- 如何沉淀团队共识。
- 如何把共识转化为 Agent 可理解的上下文和约束。
- 如何把复杂任务拆解给不同 Agent。
- 如何管理不同 Agent 的输入、输出和协同关系。
- 如何让每个 Agent 在独立边界内执行。
- 如何对不同 Agent 的产物做统一验收。
- 如何在冲突、质检失败或目标偏移时，把问题重新反馈到协作链路里。

## 5. 可能的产品方向：AI Coding 协作管理工具

为了解决上述问题，可以考虑构建一个面向团队协作的 AI Coding 协作管理工具。

它不是单纯的 Coding Agent，也不是传统项目管理看板，而是把团队共识、工程约束、任务拆解、执行状态、代码分支和验收机制连接起来。

一句话概括：

> 把团队共识变成 Agent 可执行的上下文和约束，把复杂需求拆成可并行执行的 Agent 任务，再通过评审和验收机制，把多个 Agent 的局部产物合成为可维护的工程交付。

## 6. 架构分层

### 6.1 共识层

共识层负责沉淀项目级、领域级和任务级上下文。

项目级上下文包括：

- 项目愿景。
- 核心业务目标。
- 整体技术栈。
- 项目目录结构。
- 代码规范。
- 设计规范。
- 通用字段和接口约定。

领域级上下文包括：

- 某个业务领域的核心目标。
- 该领域负责的模块边界。
- 与其他领域之间的输入输出关系。
- 特定领域的技术约束和数据约束。

任务级上下文包括：

- 当前任务要解决的问题。
- 当前任务涉及的文件和模块。
- 输入条件。
- 输出要求。
- 验收标准。
- 不允许修改的边界。

这些上下文不是简单堆给所有 Agent，而应该分层分发。公共信息需要共享，领域信息按需提供，任务信息只提供给相关 Agent，避免上下文污染和执行成本上升。

### 6.2 沟通层

沟通层负责把人的诉求转化为可评审、可执行的方案。

一个合理流程可能是：

1. 需求发起方先与 AI 沟通诉求。
2. AI 基于项目共识和上下文生成初步方案。
3. 系统识别方案涉及哪些领域、模块或相关方。
4. 相关 Agent 或相关人参与评审。
5. 评审后形成一份被确认的执行方案。

传统团队中的需求评审和技术评审，本质上是在补齐共识、风险和约束。AI Coding 协作工具也需要保留这种机制，只是可以通过 AI 先做初步校验、风险提示和方案整理，降低沟通成本。

### 6.3 Manager 层

Manager 层负责把确认后的方案拆解成可执行的 Agent 任务。

它需要处理：

- 将复杂需求拆成多个相对独立的子任务。
- 判断任务之间的依赖关系。
- 为每个 Agent 分发必要上下文。
- 明确每个 Agent 的输入、输出和验收标准。
- 控制每个 Agent 的文件修改边界。
- 尽量减少不同 Agent 之间的冲突。

这一层的目标不是让所有 Agent 自由发挥，而是让每个 Agent 在明确边界内完成自己的工作。

### 6.4 执行层

执行层是每个 Agent 具体完成任务的地方。

每个 Agent 可以围绕自己的任务形成一个小闭环：

```text
接收任务 -> 获取上下文 -> 制定计划 -> 执行修改 -> 自测/质检 -> 产出结果
```

在执行过程中，需要有明确的工程协作规则，例如：

- 不同 Agent 尽量在不同分支或 fork 中执行。
- 每个 Agent 遵守项目 Git 规则。
- 每个 Agent 只能修改自己负责范围内的文件。
- 如果需要跨边界修改，需要升级为协作问题。
- 人可以在执行过程中补充诉求、调整方向或 steer Agent。

### 6.5 验收层

验收层负责判断多个 Agent 的局部产物能否组成一个整体可交付结果。

验收不仅是看代码有没有生成，还包括：

- 是否通过测试。
- 是否符合代码规范。
- 是否符合目录结构和技术栈约束。
- 是否符合字段、接口和数据结构约定。
- 是否与其他模块存在冲突。
- 是否能合并到主分支。
- 是否满足最初的业务目标和验收标准。

不同 Agent 的结果可以通过 PR 的形式提交。系统对 PR 做冲突检测、自动测试、规范检查和跨模块一致性检查。如果出现冲突或验收失败，就把问题反馈回沟通层或执行层继续修正。

## 7. 进度管理与效果管理的区别

传统项目管理更多回答的是：任务做到哪一步了。

AI Coding 协作管理需要进一步回答：当前节点的产物是否达标，是否影响全局目标，是否需要返工，是否需要人工介入。

因此，系统不应该只是展示过程，而应该帮助用户看到：

- 每个 Agent 节点想完成什么。
- 它负责什么工作。
- 它依赖哪些上下文。
- 它最后产出了什么。
- 这个结果是否达到预设验收标准。
- 它是否会影响其他模块或全局目标。
- 当前是否存在卡点或风险。

## 8. 用户注意力的管理

AI Coding 协作工具不应该把所有过程信息都推给用户，否则只是把混乱从多个终端搬到一个新界面。

它需要做的是帮助用户过滤信息，让用户只在高价值节点介入。

需要用户介入的情况可能包括：

- 目标冲突。
- 上下文缺失。
- 多个 Agent 输出互相矛盾。
- 质检不通过。
- 代码或分支冲突。
- 需要业务判断或产品取舍。
- 需要跨领域协同。

其他常规执行、日志、过程文件和中间状态，系统应该自动归档、汇总和下钻展示，而不是持续打扰用户。

## 9. 总结

AI Coding 当前的关键矛盾，不是单个 Agent 写代码能力不足，而是 AI Coding 从个人原型走向团队工程交付时，缺少一套协作管理机制。

未来更重要的产品方向可能不是更强的单 Agent，而是面向团队的 AI Coding Orchestration：

- 把团队共识沉淀为上下文。
- 把工程规范沉淀为约束。
- 把复杂需求拆解为多个 Agent 可执行的任务。
- 把每个 Agent 的输入、输出、边界和验收标准显性化。
- 把多个 Agent 的局部产物合并为可维护、可复用、可上线的工程结果。

最终，AI Coding 需要从 Vibe Coding 走向 Team Coding，从“一个人驱动 AI 写出原型”，走向“一群人协同管理一群 AI，交付长期可维护的工程系统”。

## 10. 讨论过程记录

### 10.1 初始想法：从文件系统到 Agent 系统

我一开始的观察是，看了一些更前沿的 Agent 产品之后，会感觉它们预想的工作方式更远一些。

在传统项目管理或完成一件事情的过程中，人类负责规划，通过一些小工具或产品去执行，最后检验产出的结果。最终结果通常落在文件管理系统里，中间过程则是各种图形化界面、操作流程、代码或产品。

如果一些很小的事情已经可以被 Agent 处理，那么在复杂的大需求里，每个 Agent 都可以交付一个值得被管理的输出文件，或者一个可用性比较高的产物。这时人的核心工作可能不再是驱动单个 Agent 去执行，而是编排和处理多个 Agent。

这意味着可能需要一个类似看板的工作来管理每个节点在做什么。

第一层呈现给用户的应该是：

- 每个 Agent 的输入是什么。
- 每个 Agent 的输出是什么。
- 每个 Agent 和其他 Agent 之间的协同关系是什么。
- 用户只需要关注最终结果。

第二层才是：

- Agent 中间发生了什么。
- 哪些信息是关键步骤。
- 哪些过程可以通过图形化或产品化能力帮助用户快速找到核心信息。

第三层才是：

- Agent 交付的核心产物。
- Agent 过程中的中间文件。
- 更细粒度的执行记录。

当时的直觉是：过去人是在管理文件系统来实现事情，未来是不是可以变成管理 Agent 系统来实现事情。

### 10.2 第一轮追问：管理 Agent 系统到底创造什么价值

追问：

> 你说“人不再管理文件系统，而是管理 Agent 系统”，但这里可能混了两个层次：文件系统是结果的存放方式，Agent 系统是过程的组织方式。如果只是把多个 Agent 的产物像文件一样排列出来，那它未必比 Notion、飞书项目、GitHub Projects 更有新价值。
>
> 这个变化到底创造了什么新价值？是让人更快知道“项目现在卡在哪里”，还是让人更少介入过程，还是让多个 Agent 之间可以自动协同，还是让最终产物质量更可控？
>
> 如果没有这个 Agent 看板，用户今天在 Coding Agent 里完成一个复杂任务时，最痛的损耗到底发生在哪里？是上下文丢失、任务状态不可见、多个会话难以同步、产物难验收，还是不知道什么时候该人工介入？
>
> 可以先回答：从文件系统到 Agent 系统，真正的价值不是管理产物，而是管理____，因为在复杂 Agent 任务中，用户最大的损耗是____。

我的回答：

> 真正的价值其实不是管理产物，而是管理最后的效果。中间最大的损耗往往是没有先看清楚全局，以及对用户注意力的损耗。

### 10.3 第二轮追问：什么叫管理最后的效果

追问：

> “管理最后的效果”这个说法很有潜力，但现在还偏抽象。你得把“效果”从一个感觉词，拆成可以被产品承接的对象。
>
> 在 Coding Agent 的复杂任务里，“最后的效果”到底指什么？是最终代码能跑、功能符合需求、视觉/内容符合预期、业务指标可验证，还是“多个子任务合在一起没有冲突”？不同答案会对应完全不同的产品形态。
>
> 你说用户注意力有损耗，那用户的注意力具体被什么吃掉了？是被迫盯着日志、频繁切换会话、记住每个 Agent 的上下文、检查一堆中间文件，还是反复判断“现在该不该打断它”？
>
> 可以先补完这句：我说的管理最后效果，具体是管理____；用户注意力损耗主要来自____，所以产品不应该只展示过程，而应该帮助用户____。

我的回答：

> 最后的效果具体是当前任务最后有没有被很好地解决掉。
>
> 注意力的损耗往往来自于人要不停推动每个 Agent 执行任务，然后校验它的结果。这个过程往往伴随着很多界面切换。
>
> 所以在展示层上，应该把多个任务聚合起来；在上下文层面，应该让这些 Agent 共享同一份上下文。
>
> 这个上下文还可以分层：为了实现目标大家一定要周知的信息是一层；不同任务运行时各自需要的上下文是另一层。不同模块需要的上下文信息是有差异的。
>
> 管理最后的效果，具体管理的是复杂 Agent 是否能够实时展示项目实施进度，以及最终完成效果的状态。
>
> 用户注意力的损耗主要来自不同界面之间的切换。所以产品不应该只展示过程，而应该帮助用户一览整个项目全貌，并快速校验当前哪个项目节点是卡点。

### 10.4 第三轮追问：进度管理和效果管理有什么区别

追问：

> 这里有一个关键分叉：你现在把“进度”和“效果”放在一起讲了，但别人可能会追问：项目管理工具早就能看进度，那你这个 Agent 工作台的新东西是什么？
>
> 你说的“效果管理”相比“进度管理”多了什么？比如进度管理只回答“做到哪一步了”，效果管理是不是要回答“这一步产物是否达标、是否影响全局目标、是否需要返工”？如果是，那达标标准从哪里来，是用户提前定义、系统自动生成，还是 Agent 自评加人工抽检？
>
> 共享上下文是不是一定是好事？如果所有 Agent 都共享一份大上下文，会不会导致注意力污染、角色边界模糊、执行成本升高？你需要不要定义“公共上下文”和“私有上下文”的同步规则，比如哪些信息必须广播，哪些只在节点内保留，哪些卡点才升级给用户？
>
> 可以先回答：进度管理回答____；效果管理回答____。共享上下文不是全量共享，而是____。

我的回答：

> 在不同工作状态和任务分发下，每一个子环节本身又是一个单独完整的 loop。
>
> 这个 loop 包括：分配任务、获取必要的上下文信息、进行执行、完成工作、质检结果，以及判断这个结果怎么样。
>
> 这样才能在数据看板层面向用户传递：这个 Agent 节点到底想完成什么事情，负责什么工作，最后产出的结果是什么样的，以及在它的评估标准当中是否足以标记为完成，或者是否实现了最初的目标。

### 10.5 第四轮追问：评估标准从哪里来，什么时候打扰用户

追问：

> 你现在已经把“Agent 节点”定义成一个自带小闭环的工作单元了。下一步要逼清楚：这个闭环里的“质检标准”是谁给的，否则它很容易变成 Agent 自说自话地标记“完成”。
>
> 每个节点的评估标准从哪里来？是用户在总目标里提前定义，系统把总目标自动拆成子验收标准，还是每个 Agent 根据任务自己生成标准后交给用户确认？如果标准没有经过用户确认，它凭什么代表“最后效果达标”？
>
> 用户到底什么时候需要介入？如果每个节点都把“执行过程、质检结果、产物文件、风险提示”推给用户，那注意力损耗并没有减少，只是换了一个界面。什么级别的问题才值得打扰用户：目标冲突、上下文缺失、质检不通过、需要业务判断，还是多个 Agent 输出互相矛盾？
>
> 可以先回答：节点评估标准应该来自____；用户只应该在____时介入，否则系统应该____。

这一轮之后，我暂停了追问，开始完整整理自己的思路。

### 10.6 完整论述：从个人 AI Coding 到团队 AI Coding

我进一步整理后的想法是：

作为产品使用 AI Coding 时，遇到的痛点是：有了 AI Coding 能力之后，对 Coding 这件事情本身的认知其实没有跟上。

因此，AI Coding 最终产物难免和研发真正想要的结果不能完全对应。很多时候，它只能停留在原型阶段，而不能作为最终被上线交付的产物。这个过程会存在重复开发。

特别是对于一些复杂功能模块，需要开很多 Agent 去完成操作时，这个问题会更加明显。真实世界里，Coding 只是其中一部分；企业需要的是长期稳定、可维护的系统。

产品自己做 Vibe Coding，往往只能关注局部功能实现，但不能解决更大范围的全局效率问题。

因此，痛点是：人和 AI，或者人和一群 AI 协作处理复杂任务时，需要有人补充或补足 know-how。

这部分 know-how 在传统流程中，通常是通过开会、对齐、协作、需求评审、技术评审等动作来补全的。这也是评审机制的意义所在。

所以可以区分：

- Vibe Coding 是把自己的事变成一个想法或原型。
- Team Coding 是把团队的事变成一个工程可复用的东西。

工程可用包含很多隐性条件，包括公司或项目的代码习惯、可靠性、可维护性，以及它和其他项目之间可能存在的关联。

### 10.7 解决方向：从一个人管 AI 到一群人管一群 AI

为了解决这个问题，需要一个 AI Coding 协作管理工具。

它要把“一个人管一群 AI”，变成“一群人管一群 AI”。

前置需要共同沉淀目标共识，包括：

- 项目的大愿景。
- 每个子领域最核心要解决的问题。
- 不同领域之间输入和输出的要求。
- 相关领域或交叉领域在输出上如何协同。
- 字段结构、目录结构、技术栈、设计规范等约束。

这套工具的架构可以分成几层。

第一层是通用上下文层：

- 项目级上下文。
- 子领域级上下文。
- 任务级上下文。

这些上下文本质上都是共识，是对项目背景、任务背景、输出约束的补充。

第二层是沟通层：

- Coding 发起方先和 AI 沟通诉求。
- AI 产出初步方案。
- 如果方案涉及相关方，需要进入评审环节。
- 可以先让 AI 基于已有共识和约束进行评审。
- 更重要的方案仍然可能需要人来 review。
- 评审后的方案再反馈给 AI。

第三层是 Manager 层：

- 把方案路由和拆解成不同任务。
- 不同 Agent 执行不同任务。
- Manager 负责分配工作。
- Manager 保证每个领域尽可能相对独立。
- Manager 保证每个 Agent 都符合固定流程。

第四层是执行层：

- 每个 AI 按照项目目标进行计划和执行。
- 执行时要遵守一定的 Git 规则。
- 不同 AI 最好在不同分支或 fork 中执行，避免互相修改文件。
- 执行过程中，人可以对这些 AI 进行 steer，补充诉求或调整方向。

第五层是验收层：

- 不同 AI 形成不同 Result。
- 这些 Result 可能在不同分支上。
- 结束后通过 PR 合并成完整文档、方案或代码。
- 如果 PR 出现冲突，就说明验收有问题。
- 这个时候需要专门的人或机制处理验收问题。
- 处理结果可能以评论或反馈的方式再次返回沟通层，进入下一轮修正。
